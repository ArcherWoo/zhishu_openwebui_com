## 1. 本轮更新摘要

这次和内网部署最相关的更新，主要有下面几项：

### 1.1 `start.py` 默认更适合内网环境

- 启动默认端口已经调整为 `4173`
- 启动时默认补齐了 `NLTK_DATA`、`AUTO_DOWNLOAD_NLTK=False`
- 启动时默认补齐了 PowerPoint 离线解析相关环境变量
- 启动安装 Python 依赖时，会额外检查 `ppt_offline_runtime/wheels/`，优先从本地离线 wheel 安装

当前默认值包括：

```text
NLTK_DATA=<项目根目录>\nltk_data
AUTO_DOWNLOAD_NLTK=False
PPT_CONVERTER_COMMAND=soffice
PPT_CONVERTER_TIMEOUT_SECONDS=120
PPT_MARKITDOWN_TIMEOUT_SECONDS=120
PPT_FALLBACK_ENABLED=True
```

### 1.2 上传解密链路已经调整

现在文件上传时，如果启用了自动解密：

1. 先把原始上传文件落到本地临时文件
2. 调用 `decrypt/client.py` 里的 `decrypt_file()`
3. 只把“解密后的真实文件”上传到最终存储
4. 后续解析、向量化、问答，全部基于解密后的文件继续

这次修正后，重点变化是：

- 不再把“加密前的原文件”和“解密后的文件”重复上传两份到最终存储
- 最终入库文件名会优先使用解密后的真实文件名和扩展名
- 这对 `.ppt` / `.pptx` / `.pdf` 这类依赖后缀和 MIME 的解析链路尤其重要

### 1.3 新增 PowerPoint 离线解析链路

本轮新增了 PowerPoint 专用 loader，走的是“本地离线解析”思路，不依赖外部网络服务：

- `.pptx`：直接走 `MarkItDown`
- `.ppt`：先通过 `LibreOffice/soffice` 转成 `.pptx`，再走 `MarkItDown`
- 如果主链路失败，会退回到 `python-pptx` 文本 fallback

## 3. 当前推荐的内网部署顺序

下面这套顺序，是这次改动后推荐的实际部署顺序。

### 第 1 步：拉代码

把最新代码拉到内网服务器。

这部分会包含：

- `start.py` 的新逻辑
- 上传解密链路修正
- PowerPoint loader 代码
- 文档说明

### 第 2 步：补齐离线资源

把下面这些资源带入内网并放到项目根目录：

- `nltk_data/`
- `embedding_model/`
- `ppt_offline_runtime/`

说明：

- `nltk_data/`：给 NLTK 和部分文档解析使用
- `embedding_model/`：给本地 embedding / sentence-transformers 使用
- `ppt_offline_runtime/`：给 PPT / PPTX 离线解析使用

### 第 3 步：安装 PowerPoint 离线运行时

先进入项目根目录，再处理 `ppt_offline_runtime/`。

你至少要确保两件事成立：

1. `ppt_offline_runtime/wheels/` 存在且内容完整
2. 服务器上存在 `soffice`

其中：

- `.pptx` 主要依赖 `MarkItDown`
- `.ppt` 额外依赖 `LibreOffice/soffice`

也就是说，两者是“部分解耦”的：

- 如果 `LibreOffice` 没装好，`.pptx` 仍然可以走 `MarkItDown`
- 只有 `.ppt` 会因为缺少 `soffice` 而失败

### 第 4 步：接好解密服务

你需要在 [decrypt/client.py](C:/Users/ArcherWoo/Desktop/open-webui-main/open-webui-main/decrypt/client.py) 里实现公司的解密调用逻辑。

上传时会自动调用：

- `decrypt_file(input_path, output_path, metadata)`

同时要配置好：

- `ENABLE_UPLOAD_DECRYPTION=True`
- `DECRYPT_SERVER_URL=<公司解密服务地址>`

### 第 5 步：启动程序


## 4. PowerPoint 现在到底怎么走

这是本轮最关键的部分。

### 4.1 `.pptx` 链路

上传 `.pptx` 后，实际流程是：

1. 文件上传
2. 调用解密服务
3. 解密后的 `.pptx` 写入最终存储
4. 进入 PowerPoint loader
5. 使用 `MarkItDown` 把 `.pptx` 转成 Markdown
6. Markdown 进入切块、embedding、向量库
7. 检索时再喂给 LLM

### 4.2 `.ppt` 链路

上传 `.ppt` 后，实际流程是：

1. 文件上传
2. 调用解密服务
3. 解密后的 `.ppt` 写入最终存储
4. 进入 PowerPoint loader
5. 通过 `soffice` 把 `.ppt` 转成临时 `.pptx`
6. 再通过 `MarkItDown` 把临时 `.pptx` 转成 Markdown
7. Markdown 进入切块、embedding、向量库
8. 检索时再喂给 LLM

### 4.3 fallback 什么时候生效

如果主链路失败，例如：

- `MarkItDown` 调用失败
- 某份 `.pptx` 结构特殊
- 转 Markdown 过程中抛异常

系统会尝试走一个基础文本 fallback，把可提取的文本尽量保下来，避免整条链路直接断掉。

但是要注意：

- fallback 的目标是“尽量能入库”
- 不是“版式 100% 完美恢复”

## 5. 这次内网部署要特别注意的几个点

### 5.1 `ppt_offline_runtime/` 不是代码仓库内容

不要指望 `git pull` 后自动有它。

正确做法是：

1. 去 Release 下载 `ppt_offline_runtime.zip`
2. 解压到项目根目录
3. 再启动

### 5.2 `.pptx` 和 `.ppt` 不是同一层依赖

要分清楚：

- `.pptx` 主要看 `MarkItDown` 是否安装好
- `.ppt` 还要额外看 `LibreOffice/soffice` 是否可用

所以排查时不要混在一起。

### 5.3 `sentence-transformers` 不能随便关

这轮之前已经验证过：

- 如果本地 embedding 模型没补齐
- 或者把 sentence-transformers 相关能力关掉

虽然有些文件还能“解析”，但向量质量会明显下降，问答效果也会变差。

所以内网部署时要确保：

- `embedding_model/` 不是空的
- 配置确实指向本地可用模型

### 5.4 解密后只保留一份最终文件是正常的

这次上传逻辑修完后，目标就是：

- 原始上传文件只作为临时输入
- 解密后的真实文件进入最终存储和后续处理

如果你看到临时目录里有中间文件，这是正常的。

但最终业务处理应当围绕“解密后的文件”继续，而不是两边都重复入库。

## 7. 推荐的排查顺序

如果上传后前端一直转圈，但后台没有明显报错，建议按下面顺序查。

### 7.1 先看解密是否真的成功

确认：

- 原文件存在
- 解密后的文件存在
- 解密后的后缀是否正确

例如：

- 原文件是加密后的 `xxx.dat`
- 解密后应该恢复成真实的 `xxx.pptx` 或 `xxx.ppt`

### 7.2 再看 PowerPoint 依赖是否完整

对 `.pptx`，重点看：

- `ppt_offline_runtime/wheels/` 是否完整
- `markitdown` 是否安装成功

对 `.ppt`，重点再加一条：

- `soffice` 是否存在且命令可执行

### 7.3 再看 embedding 模型是否可用

如果文档已经能解析，但后续问答质量不对，就重点看：

- `embedding_model/` 是否完整
- 本地 embedding 配置是否正确

### 7.4 最后再看向量库和模型配置

如果文件已成功入库，但聊天时报模型问题，就查：

- 普通用户是否有可用模型权限
- 模型 ID 是否和管理员配置一致
- 外部 LLM API 地址是否可达

## 10. 补充说明

### 10.1 关于 `pyodide_runtime/`

这个目录是给前端 Pyodide 离线资源准备的中文说明入口，不等于“实际生效目录”。

真正生效的通常还是：

- `vendor/npm/`
- `static/pyodide/`

如果你短期内不使用代码执行功能，可以暂时不把它作为第一优先级。
