# 默认知识库样板设计

## 目标

为每个用户提供一个开箱即用的业务知识沉淀起点，减少“知识库空白页”带来的无从下手问题。

本次设计采用现有知识库结构，不新增新的产品模块，不新增新的前端入口。用户登录后看到的仍然是现在的“知识库”能力，只是在合适的时机自动获得一套默认样板内容。

本次需要同时满足两类场景：

- 新创建用户自动获得一套默认知识库样板
- 现有用户补发一套默认知识库样板，便于先试用和验证效果

## 设计结论

采用“方案 1”：

- 每个用户拥有一个默认知识库
- 默认知识库名称为 `部门业务知识沉淀样板库`
- 默认知识库下预置 3 篇模板文本
- 模板文本全部复用现有“添加文本内容”对应的文件与知识库处理链路
- 用户可自行修改、删除、重命名，不做特殊限制

本次暂不实现“公共知识库”。公共知识库是下一阶段需求，避免和“每人一份可自由修改的私有样板”混在一起，造成权限和认知混乱。

## 范围

本次覆盖：

- 新用户注册时自动初始化默认知识库样板
- 管理员后台新增用户时自动初始化默认知识库样板
- 为现有用户批量补齐默认知识库样板
- 默认知识库与默认模板文案设计
- 幂等控制，避免重复创建

本次不覆盖：

- 公共知识库
- 样板内容的版本升级与自动迁移
- 用户删除样板后自动重建
- 多套部门模板切换
- 新的知识库模板管理后台

## 核心原则

### 1. 完全复用现有知识库结构

默认样板必须是“普通知识库 + 普通文本文件”，这样用户的后续使用体验、搜索逻辑、RAG 行为、权限模型都和现有系统完全一致。

### 2. 样板要易上手，不要像制度文件

默认内容不是为了展示系统能力，而是为了帮业务同事“迈出第一步”。模板内容要同时满足：

- 看得懂
- 能照抄
- 能直接改
- 足够完整
- 不会太学术化

### 3. 默认创建必须幂等

无论是新用户自动初始化，还是现有用户批量补发，都不能重复生成多套相同样板。

### 4. 用户拥有完全控制权

样板创建后就是用户自己的知识库。用户可以：

- 改知识库名
- 改文档标题
- 改内容
- 删除文档
- 删除整个知识库

系统不做“强制保留”。

## 方案设计

## 一、默认知识库信息

每个用户默认获得一个知识库：

- 名称：`部门业务知识沉淀样板库`
- 描述：`用于沉淀部门业务流程、经验方法、常见问题和协作要点的起步样板，可直接修改后投入使用。`

该知识库对用户本人表现为普通私有知识库，不额外增加特殊 UI。

建议在 `meta` 中打内部标记，便于后续幂等判断与排查，例如：

```json
{
  "seeded_by_system": true,
  "seed_template_key": "department-knowhow-starter",
  "seed_template_version": 1
}
```

该标记仅用于系统内部识别，不在前端新增展示。

## 二、默认模板文档设计

默认知识库下预置 3 篇文本。

### 文档 1：业务知识 Knowhow 沉淀说明与范文

文档名：

`01-业务知识Knowhow沉淀说明与范文`

定位：

- 告诉用户“为什么要沉淀”
- 告诉用户“什么内容值得沉淀”
- 给一篇完整范文，让用户知道成品应该长什么样

建议结构：

1. 什么是业务知识 Knowhow
2. 哪些内容适合沉淀
3. 写得有用的 5 个原则
4. 一篇完整示例

示例内容建议选一个所有部门都容易理解的通用场景，例如：

- 客户需求受理
- 订单异常处理
- 跨部门协作交接

范文不追求行业绝对准确，而是追求结构清晰、表达接地气。建议包含：

- 场景背景
- 适用范围
- 处理流程
- 判断标准
- 易错点
- 沟通建议
- 一个典型案例

## 三、创建时机设计

### 1. 新用户自动初始化

在用户创建成功后执行默认知识库初始化。

覆盖场景：

- 普通注册用户
- 管理员后台新增用户
- 未来若有批量导入新增用户，也应复用同一初始化函数

推荐做法：

- 抽一个统一初始化函数，例如 `ensure_default_knowledge_templates_for_user(user_id, ...)`
- 在用户创建成功后调用
- 如果初始化失败，不回滚用户创建，但要记录明确日志

理由：

- 用户创建成功比知识库初始化更基础
- 避免因为样板初始化失败，导致账号都创建失败
- 运维排查时更容易定位问题

### 2. 现有用户补发

提供一个一次性补发入口，为当前已有用户补齐默认知识库样板。

推荐实现方式：

- 后端提供一个管理员可触发的补发逻辑
- 逻辑遍历现有用户并调用同一个 `ensure_default_knowledge_templates_for_user(...)`

不建议把补发逻辑写成前端脚本或手工 SQL，因为：

- 需要同时创建知识库、文本文件、知识库文件关联
- 还要触发后续解析与向量化链路
- 手工操作容易漏步骤

## 四、幂等策略

默认样板初始化必须满足“多次执行，结果仍然只有一套”。

建议规则：

### 1. 知识库级别

优先按 `meta.seed_template_key == "department-knowhow-starter"` 判断是否已存在默认知识库。

不建议只靠名称判断，因为用户可能会改名。

### 2. 文档级别

在默认知识库下，对每篇模板文档打对应标记，例如：

```json
{
  "seeded_by_system": true,
  "seed_template_key": "department-knowhow-starter",
  "seed_document_key": "knowhow-guide"
}
```

这样补发时可以做到：

- 缺整个知识库时：创建整套
- 知识库在但缺某篇模板时：只补缺的
- 文档已存在且用户已修改时：不覆盖

### 3. 删除行为

如果用户删除了默认知识库或默认文档，系统不自动重建。

理由：

- 尊重用户控制权
- 避免出现“删了又回来”的反感体验

## 五、实现边界

本次只复用现有结构，不引入新的产品概念。

推荐新增的代码形态应尽量是“内部服务函数”，而不是新产品功能。例如：

- 一个默认样板内容模块
- 一个默认知识库初始化模块
- 一个现有用户补发入口

不需要新增：

- 知识库模板管理页面
- 模板商城
- 模板开关中心

## 六、推荐代码组织

### 1. 默认内容定义

建议新增一个专门保存默认文案的后端模块，例如：

- `backend/open_webui/utils/default_knowledge_templates.py`

职责：

- 返回默认知识库名称、描述
- 返回 3 篇模板文档标题与正文
- 返回模板版本号与内部 key

这样后续如果你要继续增加“公共知识库”或“部门专属模板”，可以在这里继续扩展，而不会把文案散落到用户创建逻辑里。

### 2. 初始化服务

建议新增一个专门的服务模块，例如：

- `backend/open_webui/utils/default_knowledge_initializer.py`

职责：

- 检查用户是否已拥有默认样板
- 创建默认知识库
- 创建默认模板文本
- 将模板文本关联到知识库
- 触发后续现有处理链路

### 3. 调用点

建议调用点包括：

- 用户创建成功后的统一后置逻辑
- 管理员批量补发逻辑

## 七、模板内容风格要求

为了让一线同事愿意用，模板文案必须满足下面几个标准：

### 1. 不写空话

不要出现过多“赋能、闭环、协同机制、最佳实践沉淀”等空泛表达。

### 2. 一看就知道怎么抄

每篇模板都要让用户明白：

- 这篇是拿来干什么的
- 我从哪里开始改
- 我应该替换哪些内容

### 3. 保持结构稳定

三篇模板之间不要完全不同风格。标题层级、语言习惯和字段形式应尽量统一，降低学习成本。

### 4. 同时适合人看与模型检索

文案要适合知识库检索，因此每篇模板都要包含：

- 场景词
- 流程词
- 角色词
- 异常词
- 术语词

## 八、验收标准

满足以下条件即可认为本次设计落地成功：

- 新注册用户首次进入知识库时，能看到 `部门业务知识沉淀样板库`
- 管理员后台新增的用户，也会自动获得同样样板
- 现有用户可以被补发一套样板
- 每个用户默认样板中包含 3 篇文本模板
- 模板内容同时包含说明、范文、填空模板和 FAQ/易错点模板
- 重复执行初始化不会创建多套相同样板
- 用户修改、重命名、删除后，系统不强制恢复

## 九、后续阶段

本次完成后，下一阶段再单独讨论“公共知识库”。

公共知识库与本次默认私有样板的目标不同：

- 默认私有样板：帮助每个人开始沉淀
- 公共知识库：让大家共享已沉淀成果

这两个能力应该分阶段建设，避免第一次就把“模板起步”和“共享治理”揉在一起，导致权限、维护责任和用户认知都变复杂。
