# 知识库 Markdown 工作台设计

## 目标

把知识库里的 Markdown 文件查看体验，从“左上角堆内容的简陋预览框”升级成“可阅读、可编辑、可自动保存的文档工作台”。

这次设计重点解决四个问题：

- Markdown 打开后版面失衡，内容挤在左侧，右侧空白浪费严重
- 表格模板在当前预览中极难阅读，尤其不适合采购类台账文档
- 预览和编辑完全分离，切换割裂，不符合“边看边改”的使用习惯
- 自动保存和保存状态缺位，用户改文档时心里没底

## 设计结论

采用“连续文稿纸面 + 嵌入式宽表格台账区 + 块级原地编辑 + 自动保存状态条”的方案，并且只对知识库中的 Markdown 文件启用这套新体验。

普通文字内容按成文文档方式连续阅读，表格内容按嵌入正文的业务台账方式呈现；用户默认看到的是渲染后的成品页面，点击某个内容块后可在原位置直接编辑，不再强制在“预览/编辑”两个全页模式之间来回切换。

视觉上不再让每个块都像独立卡片，而是把整篇 Markdown 收成一张文稿纸面：正文块默认几乎不显边界，只在 hover / focus 时给出很轻的编辑提示；表格块保留更稳定的容器感，但也要看起来像文稿中的一个嵌入区域，而不是另一个漂浮面板。

## 范围

本次覆盖：

- 知识库中 `.md` / `.markdown` / `.mdx` 文件的查看体验升级
- 顶部文档工具条与明确的保存状态反馈
- 居中文档画布与宽内容区域的混合布局
- Markdown 块级原地编辑
- 表格块的专门阅读样式与结构化编辑态
- 自动保存、防抖保存、失败提示、离页保护
- 仅在知识库文件抽屉内生效，不新增新的路由页面

本次不覆盖：

- 非 Markdown 文本文件的全面重做
- 真正的富文本所见即所得编辑器
- Excel 级别的表格能力，例如批量粘贴、多选区域、公式、拖拽列宽
- 多人协同编辑
- 文档版本历史、评论、审批流

## 一、页面定位

这页不再被定义为“文件预览”，而是“知识文档工作台”。

它要同时满足两类场景：

- 阅读场景：用户打开模板、范文、说明文档时，能舒服地看
- 修改场景：用户想把模板改成自己的内容时，能顺手地改

因此它的首要原则不是“多功能”，而是“阅读和编辑发生在同一个上下文里”。

## 二、页面结构

### 1. 顶部文档工具条

知识库 Markdown 打开后，文件抽屉顶部改成稳定的文档工具条，包含：

- 返回按钮
- 文件名与文件类型标识
- 保存状态：`已保存` / `保存中...` / `保存失败`
- 次级操作：复制 Markdown、下载、更多菜单

如果文件为只读，则不显示可编辑提示，但仍保留阅读版式。

### 2. 文档画布区

正文区域采用居中的单张文档纸面，而不是让内容贴着左边展开，也不是把每一段拆成一张张独立卡片。

页面分为两种内容宽度：

- 主文档列：用于标题、段落、列表、引用、普通说明，宽度以舒适阅读为目标
- 宽内容区：用于表格、代码块、较宽的结构化内容，允许突破主列但仍保持整体居中

这样可以同时兼顾“文档感”和“台账感”，并且让整页在视觉上更像“一篇东西”。

### 3. 右侧不做常驻功能栏

本次不增加右侧常驻属性栏或评论栏，避免把阅读区再次挤窄。右侧空间优先用于给主文档留白，形成更稳的视觉中心。

## 三、阅读体验设计

### 1. 普通文档内容

标题、段落、列表、引用、分割线等内容，按文档阅读方式排版：

- 标题层级清晰
- 正文行高舒适
- 小节之间有明确留白
- 列表与引用有足够缩进与边界
- 不让所有元素堆成一个密集文本块

### 2. 表格内容

采购类和台账类模板的核心问题是表格过于拥挤，因此表格必须单独处理：

- 表格进入宽内容容器
- 表头样式增强
- 单元格默认左上对齐
- 长文本允许换行，避免挤成极窄竖列
- 使用轻边框、行分隔和柔和背景，提升可扫读性
- 表格整体允许横向滚动，但滚动容器本身要清晰、整洁

### 3. 内容节奏

正文、表格、说明文字之间需要用留白、轻分隔和局部浅底色建立节奏，而不是靠厚重卡片和更多边框堆砌。目标是让用户一眼分清：

- 这是文档说明
- 这是待填写模板
- 这是可维护台账

## 四、编辑模型

### 1. 不再保留全页“预览 / 编辑”二选一

对 Markdown 文件，顶部不再使用当前那种全页级的 `Preview / Edit` 切换。

改成：

- 页面默认显示渲染后的文档
- 用户点击某个块后，在原位进入编辑
- 编辑完成后，自动保存并回到渲染状态

这能保留阅读体验，也让修改动作更自然。

### 2. 块级原地编辑

块级编辑是这次方案的核心。

块的粒度定义为顶层 Markdown 结构块，例如：

- 标题
- 段落
- 列表
- 引用
- 代码块
- 表格

编辑行为如下：

- 默认态：显示渲染结果
- Hover 态：给当前块极轻的提示，表达“这里可以修改”，但不破坏整篇文稿的连续感
- 激活态：点击某一块后，该块切换为编辑态，其它块保持渲染态
- 完成态：失焦或保存成功后，该块回到渲染态

Markdown 仍然是唯一真实数据源，不引入第二套富文本存储格式。

### 3. 表格块的特殊编辑态

表格块不采用普通段落的单个文本框，而是进入结构化编辑态：

- 表头可见
- 单元格可直接修改
- 当前行高亮
- 支持新增一行、删除一行、复制一行
- 保存时重新生成 Markdown 表格文本

本次只做“够用的结构化表格编辑”，不追求 Excel 级交互。

### 4. 默认边界策略

为了避免页面看起来像“很多块拼起来”，块级边界策略明确如下：

- 普通正文块：默认无明显边框或阴影
- 普通正文块：hover / focus 时再出现极淡底色或细描边
- 表格块：默认保留轻容器，保证稳定性与可扫读性
- 编辑态：当前块再明确高亮，和默认态拉开层次

也就是说，页面要优先像文章，编辑提示是第二层信息，而不是第一层信息。

## 五、自动保存设计

自动保存为默认策略，不要求用户频繁点保存按钮。

保存策略：

- 内容发生变化后，采用短时间防抖自动保存
- 切换块或点击其他区域时，立即触发一次保存
- 页面离开前，如果还有未完成保存，要进行拦截或强制 flush

状态反馈：

- 顶部工具条常驻保存状态
- 保存成功后静默显示“已保存”
- 保存中显示“保存中...”
- 保存失败时在顶部和当前块附近都给出明确提示

失败时要求：

- 当前本地编辑内容不能丢
- 用户可以继续修改
- 用户可以重试

## 六、文件类型策略

这次新工作台只对 Markdown 文件启用：

- `.md`
- `.markdown`
- `.mdx`

其它文本文件继续沿用当前的简单编辑模式，避免一次性把所有文件预览场景都重做，扩大风险。

这也意味着：

- 新生成的模板与范文优先使用 Markdown
- 历史遗留 `.txt` 文件可继续打开，但不强制迁移

## 七、技术方案

### 1. 组件拆分

当前 [KnowledgeBase.svelte](C:/Users/ArcherWoo/Desktop/open-webui-main/open-webui-main/src/lib/components/workspace/Knowledge/KnowledgeBase.svelte) 已经较大，本次不建议继续把全部逻辑堆进去。

建议拆成以下几个聚焦单元：

- `KnowledgeFileWorkbench.svelte`
  - 负责 Markdown 文件工作台整体壳层、工具条、保存状态与模式判断
- `KnowledgeMarkdownDocument.svelte`
  - 负责按块渲染整篇文档
- `KnowledgeMarkdownBlock.svelte`
  - 负责通用块的渲染态与编辑态切换
- `KnowledgeMarkdownTableEditor.svelte`
  - 负责表格块的结构化编辑
- `knowledgeMarkdownBlocks.ts`
  - 负责把 Markdown 内容拆成可编辑块，并在保存后重新拼装
- `knowledgeMarkdownTables.ts`
  - 负责 Markdown 表格与结构化表格数据之间的转换
- `knowledgeMarkdownEditorState.ts`
  - 负责保存状态、脏数据判断等纯状态逻辑

### 2. 数据流

整体数据流保持现有接口不变，只改前端状态组织：

1. 知识库抽屉拿到原始 Markdown 文本
2. 工作台把文本解析成块列表
3. 页面按块渲染
4. 用户只修改某一个块
5. 前端把该块内容回写为新的整篇 Markdown
6. 自动保存调用 `updateFileDataContentById(...)`
7. 保存成功后刷新本地快照与保存状态

### 3. 表格数据流

表格块额外增加一层转换：

1. Markdown 表格文本解析成列定义与行数据
2. 用户按单元格修改
3. 编辑完成后重新生成标准 Markdown 表格文本
4. 再合并回整篇文档

## 八、错误处理

需要覆盖四类错误：

### 1. 内容为空

- 页面展示空态，而不是一片白板
- 如果文件可编辑，给出清晰的“开始填写”入口

### 2. Markdown 解析异常

- 不因某一个块的解析失败导致整篇页面崩掉
- 失败块退回原始文本编辑态

### 3. 自动保存失败

- 保留本地草稿
- 顶部状态变为失败
- 当前块显示失败提示

### 4. 表格转换失败

- 不丢原始 Markdown 表格文本
- 回退到原始文本编辑

## 九、可访问性与易用性

需要保证：

- 所有可编辑块都有明确 hover / focus 态
- 键盘可以进入编辑与退出编辑
- 保存状态文本可被屏幕阅读器识别
- 表格编辑中的输入框有明确列名或标签

## 十、验收标准

满足以下条件即可认为本次设计落地成功：

- 打开知识库中的 Markdown 文件后，正文不再挤在左上角，页面有清晰的居中文档画布
- 表格模板在视觉上明显比当前版本更易读，尤其是采购台账类模板
- Markdown 文件不再依赖全页 `Preview / Edit` 切换，用户可在渲染页面中原地编辑
- 普通块支持原地编辑并自动保存
- 表格块支持结构化编辑并能正确回写 Markdown
- 顶部有明确保存状态，失败时有清晰提示
- 非 Markdown 文本文件仍可继续使用，不受本次改动影响
- 现有后端文件保存接口无需改动即可支撑整个流程
